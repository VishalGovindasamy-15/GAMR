I think the project has evolved a lot from where it started. Originally, it was inspired by GTA 6's asset streaming. Now it's become something much broader.

One important distinction before I summarize it:

* **The core runtime idea is technically plausible.**
* **Some of the future capabilities (such as automatically improving training accuracy or outperforming all existing systems) are still hypotheses that would need experimental validation.**

With that in mind, here's how I would define the project today.

# Research Results So Far

Your experiments suggest:

* SSD latency is stochastic.
* PCIe transfers are comparatively stable.
* GPU execution for the tested kernels is much faster than storage transfer.
* Storage behavior, not GPU compute, is the dominant uncertainty in the measured setup.

Those results shaped the current architecture.

# Proof of Concept (Current Goal)

The immediate goal is **not** to build the perfect adaptive runtime.

The goal is:

1. Split model into blocks.
2. Stream blocks.
3. Produce identical outputs.
4. Reduce VRAM requirements.
5. Keep the model running correctly.

Only after that should adaptation be added.

# What Makes This Different

From everything we've discussed, I think the distinctive idea is **not simply streaming weights**‚ÄĒthat concept already exists in various forms.

The more distinctive direction is:

> **Treating the entire AI memory hierarchy (SSD, RAM, VRAM, GPU) as one adaptive system that continuously observes hardware conditions, respects system resource budgets, and dynamically decides where different kinds of AI data (weights, KV cache, activations, gradients, optimizer states) should reside.**

If you can demonstrate that this adaptive manager consistently uses less VRAM, runs on diverse hardware without manual tuning, and maintains correct model execution, that would be a meaningful contribution. Whether it also improves throughput, supports much larger models, or benefits training are the next hypotheses to test once the proof of concept is working.



I actually think **Docker is the right choice**, but **not yet as the "AI researcher."**

Use Docker first as the **execution environment**, not the decision-maker.

This is an important distinction.

## My recommendation

I would absolutely use Docker **from the beginning**, but I would keep its responsibility narrow:

* **Containerize** the runtime so it runs the same way everywhere.
* **Automatically collect** logs, metrics, configurations, and outputs.
* **Make every run reproducible** so you can compare changes later.

I would **not** let Docker or an AI agent modify algorithms or formulas during the POC stage. First build a runtime that works correctly and produces complete diagnostics. Once you have that stable foundation, you can add an experiment engine that automatically launches many Docker runs with different configurations and compares the results. That gives you automation without making the core runtime harder to debug.
